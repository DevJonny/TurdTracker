## Codebase Patterns
- Build command: `DOTNET_CLI_USE_MSBUILD_SERVER=0 DOTNET_HOST_PATH=$(which dotnet) dotnet build` (required on macOS ARM64 to avoid task host error)
- `wasm-tools` workload must be installed (`dotnet workload install wasm-tools`) â€” without it, ComputeWasmBuildAssets fails with MSB4216
- Build with sandbox disabled for NuGet restore (network access required)
- Project source lives in `src/TurdTracker/`
- MudBlazor v9.2.0 configured with providers in MainLayout.razor
- Blazored.LocalStorage v4.5.0 (must specify version explicitly; latest search fails on .NET 10)
- Folder structure: Pages/, Components/, Models/, Services/ under src/TurdTracker/
- MudBlazor CSS/JS referenced in wwwroot/index.html
- Build from repo root: `DOTNET_CLI_USE_MSBUILD_SERVER=0 dotnet build src/TurdTracker/TurdTracker.csproj`
- DiaryService uses LocalStorage key "diary-entries"
- DI registration pattern: `builder.Services.AddScoped<IService, Service>()` in Program.cs
- SyncService subscribes to `DiaryService.OnDataChanged` with 2-second debounce; temporarily unsubscribes during `ReplaceAllAsync` to avoid sync loops
- SyncService handles 412 (re-download + re-merge + retry up to 3x) and 401 (silent re-auth) automatically
- Inject `ISyncService` for sync operations; call `InitializeAsync()` on app startup
- Page injection pattern: `@using TurdTracker.Services` + `@inject IDiaryService DiaryService`
- `_Imports.razor` has global `@using TurdTracker.Services` â€” no per-page using needed
- MudBlazor v9 MUD0002 warning: avoid HTML attributes like `Title` on MudBlazor components; use MudBlazor-specific props
- Client ID for Google OAuth: 101133685796-p43f4uqejgt9lgce4lvibrnoam8oegb1.apps.googleusercontent.com
- DiaryService internal reads use private `GetAllRawAsync()` (unfiltered); public `GetAllAsync()` filters soft-deleted entries
- `OnDataChanged` event pattern: fire after `SaveAsync()` in mutating methods (Add, Update, Delete)
- JS interop files live in `wwwroot/js/`; call from C# via `IJSRuntime.InvokeAsync("googleAuth.method", args)` (no `window.` prefix)
- GoogleAuthService wraps JS interop; inject `IGoogleAuthService` for auth operations
- GoogleDriveService uses HttpClient with Bearer token; inject `IGoogleDriveService` for Drive file operations
- Google Drive API: files endpoint at `https://www.googleapis.com/drive/v3/files`, upload at `https://www.googleapis.com/upload/drive/v3/files`
- Sync file name: `turdtracker-sync.json` stored in `appDataFolder` space
- MergeEngine is static â€” call `MergeEngine.Merge(local, remote)` and check `MergeResult.LocalChanged`/`RemoteChanged` to decide what to save/upload
- MainLayout.razor: sync icon is between MudSpacer and theme toggle; uses `@implements IDisposable` to clean up event subscriptions
- Use `InvokeAsync(StateHasChanged)` when handling events from services (not on Blazor sync context)
- MudBlazor v9.2.0 does not have `Icons.Material.Filled.Google` â€” use `Icons.Material.Filled.Cloud` as alternative

  - HttpClient auth header set via `DefaultRequestHeaders.Authorization` before each call

## 2026-03-24 - US-006
- Created static `MergeEngine` class in `Services/MergeEngine.cs` with `Merge(List<DiaryEntry> local, List<DiaryEntry> remote)` method
- Created `MergeResult` class with `MergedEntries`, `LocalChanged`, `RemoteChanged` properties
- Additive merge: entries only in local or only in remote are kept; entries in both resolved by latest `LastModified` (UTC)
- Soft-deleted entries included in merged result; tombstones older than 90 days are purged
- `LocalChanged` set true when remote has newer or unique entries; `RemoteChanged` set true when local has newer or unique entries
- Files changed: `src/TurdTracker/Services/MergeEngine.cs` (new)
- **Learnings for future iterations:**
  - MergeEngine is a static utility class â€” no DI registration needed
  - `LocalChanged` indicates the local store needs updating; `RemoteChanged` indicates remote (Drive) needs uploading
  - Tombstone purge uses `DateTime.UtcNow` comparison, consistent with `LastModified` being stored as UTC
  - When entries tie on `LastModified`, local wins (>= comparison) â€” this avoids unnecessary remote uploads for identical data

## 2026-03-24 - US-007
- Created `ISyncService` interface with `SyncAsync()`, `SyncStatus` property (enum: NotSignedIn, Idle, Syncing, Synced, Error), `LastSyncedUtc`, `OnSyncStatusChanged` event, `InitializeAsync()`
- Created `SyncService` implementation orchestrating: check auth â†’ download remote â†’ merge â†’ save local if changed â†’ upload if changed
- Handles 412 Precondition Failed with re-download, re-merge, retry (up to 3 attempts)
- Subscribes to `DiaryService.OnDataChanged` with 2-second debounce via `CancellationTokenSource`
- `InitializeAsync()` triggers sync on startup if user is signed in
- Silently skips sync when offline (`HttpRequestException` caught)
- On 401, attempts silent re-auth via `SignInAsync()` before setting Error status
- Temporarily unsubscribes from `OnDataChanged` during `ReplaceAllAsync` to prevent sync loops
- Implements `IDisposable` to clean up event subscription and debounce CTS
- Registered as scoped in `Program.cs`
- Files changed: `src/TurdTracker/Services/ISyncService.cs` (new), `src/TurdTracker/Services/SyncService.cs` (new), `src/TurdTracker/Program.cs`
- **Learnings for future iterations:**
  - When writing merged data back to local storage, unsubscribe from `OnDataChanged` first to avoid infinite sync loops
  - `Task.Run` with `Task.Delay` + `CancellationTokenSource` is the pattern for debouncing in Blazor WASM (no `System.Timers.Timer` needed)
  - SyncService implements `IDisposable` â€” Blazor's DI container disposes scoped services automatically
  - Check `SyncStatus == Syncing` at start of `SyncAsync()` to prevent concurrent sync operations
